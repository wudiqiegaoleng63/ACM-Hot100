# Claude Code · Post-MVP 续跑主提示词

将下面“主提示词”整体交给编码模型。一次对话只执行一个任务。

---

## 主提示词

你在 `ACMHOT100` 仓库中继续开发。MVP 已完成，现在执行 Post-MVP 上线与演进计划。

你的执行能力有限，因此必须严格缩小范围：**一次只完成一个任务，不得继续下一项。**

### 必读文件

按顺序完整阅读：

1. `CLAUDE.md`
2. `docs/MVP_DESIGN_SPEC.md`
3. `docs/NEXT_IMPLEMENTATION_PLAN.md`（这是已完成的历史计划，只用于理解上下文）
4. `docs/POST_MVP_IMPLEMENTATION_PLAN.md`（这是当前唯一活跃实施计划）

如果任一文件不存在、内容冲突或读取不完整，停止并准确报告，不要自行猜测。

### 本次唯一目标

只执行 `docs/POST_MVP_IMPLEMENTATION_PLAN.md` 中第一个未勾选的 `P5-xx` 任务。

- 如果第一个未勾选任务是 `P5-01`，只做 P5-01。
- 即使时间、上下文和 Token 充足，也不得开始 P5-02。
- Phase Gate 不是独立编码任务，但进入下一 Phase 前必须逐条检查。
- 不得提前实现 Phase 6。

### 权限边界

本轮默认权限如下：

- 允许读取和修改当前仓库内与本任务直接相关的文件。
- 允许运行本地测试、构建、Docker 和只读诊断命令。
- 允许在任务全部验证通过后创建一个本地 Git commit。
- 禁止 `git push`。
- 禁止创建远程分支、PR、Release、Package 或修改 GitHub 仓库设置。
- 禁止登录或修改服务器、DNS、SMTP、云平台和生产数据。
- 禁止执行生产部署、压测或故障注入。

只有用户在当前对话明确授权某一项外部操作，才可以扩大对应权限；历史对话中的泛化授权不能代替当前授权。

### 强制开工流程

1. 运行：

   ```bash
   git status --short --branch
   git log --oneline --decorate -5
   git diff --check
   ```

2. 确认当前分支、HEAD、相对远端状态和工作树状态。
3. 如果工作树不干净：
   - 判断改动是否属于用户或其他任务；
   - 不得覆盖、stash、reset、checkout 或删除；
   - 能安全避开才继续，否则停止并列出冲突文件。
4. 从计划原文抄出本任务的：目标、前置、只做、不得做、验证、人工确认、建议提交信息。
5. 检查前置任务与 Phase Gate。未满足就停止，不得越过。
6. 只阅读与本任务相关的代码和测试，不进行全仓库重构。

### 官方文档规则

编码前必须查本任务涉及技术的最新官方文档：

- GitHub 功能只用 GitHub Docs 和相关官方 Action 仓库。
- Docker/Compose 只用 Docker 官方文档。
- Go 标准库与 Go 工具优先 go.dev、pkg.go.dev 和官方仓库。
- React/Vite/测试库使用各自官方文档。
- Judge0 使用 Judge0 官方仓库、配置和 API 文档。
- OpenAPI 使用 OpenAPI Initiative 规范。
- MySQL/Redis/Caddy/Prometheus 使用各自官方文档。

不要依赖博客、搜索摘要或过时教程做安全和生产配置。将实际查阅的链接写入最终汇报。

### 实现规则

1. 严格限于本任务“只做”列表。
2. 明确遵守“不得做”列表。
3. Bug、安全策略、状态机和业务逻辑：先写会失败的测试，再修改实现。
4. 配置和部署文件：先写可自动执行的语法/静态校验，再补人工验收步骤。
5. 不允许使用 `any`、`@ts-ignore`、空 catch、硬编码成功响应或无条件 retry 掩盖问题。
6. 不允许删除、skip、弱化断言或放宽安全策略来获得绿色结果。
7. 不引入新的技术栈，除非本任务明确要求；引入依赖前说明为什么现有能力不足。
8. 新依赖必须锁版本；GitHub Actions 必须按计划固定完整 commit SHA。
9. 不记录或提交真实 Secret、Token、Cookie、用户源码、隐藏测试输入或生产个人信息。
10. 外部资源缺失时，不得用 Mock 冒充真实验收。
11. 遇到计划外缺陷，只记录，不顺手修复；只有它直接阻塞本任务且改动最小才可处理，并在汇报中说明。

### 外部资源闸门

任务涉及以下内容时，先检查计划第 5 节的 G0–G7：

- GitHub push、Actions 真实运行、Packages；
- 云服务器、域名、DNS、防火墙；
- SMTP 和发信 DNS；
- 异机备份存储；
- 独立 Judge0 节点；
- 生产开放；
- 100 题内容授权。

缺少用户才能提供的资源时：

- 只完成任务明确允许的仓库内准备；
- 如果任务的完成定义要求真实外部证据，则不得勾选为完成；
- 按“仓库内完成待外部验收”汇报并停止；
- 列出变量名、权限和资源类型，不索要用户把明文 Secret 发进代码或计划文件。

### 验证规则

1. 运行任务列出的全部验证命令，不得省略失败命令。
2. 代码任务还必须运行计划第 11 节固定验证矩阵。
3. 配置/Docker/CI 任务必须执行能在本地安全运行的语法和构建验证。
4. 记录每条命令的真实 exit code 和关键结果。
5. 运行：

   ```bash
   git diff --check
   git status --short
   git diff --stat
   ```

6. 检查 diff 没有无关改动、生成物、Secret 和调试代码。
7. 任一必需验证失败，不得勾选计划、不得提交“完成”。

### Git 规则

全部验证通过且人工验收要求已满足后：

1. 只将当前任务从 `[ ]` 改为 `[x]`。
2. 计划勾选与代码放在同一个提交。
3. 使用计划给出的建议提交信息；必要时只能做小幅准确调整。
4. 创建一个本地 commit。
5. 提交后运行：

   ```bash
   git status --short --branch
   git log -1 --oneline
   ```

6. 禁止 push。

如果只完成仓库内准备但缺少外部真实验收，不勾选任务；是否允许提交准备工作，以任务说明和用户当轮指令为准。默认不创建“半完成”提交。

### 允许停止的阻塞条件

只有以下情况可以停止而不提交：

- 缺少计划 G0–G7 中只有用户能提供的资源或授权；
- 当前工作树改动会与本任务冲突，继续会覆盖用户工作；
- 官方文档与计划存在会改变安全边界的冲突；
- 必需工具或外部服务不可用，且已完成安全、最小的本地诊断；
- 连续三次针对同一根因的安全修复仍失败。

停止时必须给出：准确错误、已运行命令、相关文件、根因判断、需要的最小用户输入。禁止只写“环境问题”。

### 固定最终汇报

严格使用以下格式，然后立即停止：

```text
任务：P5-xx <名称>
结论：完成 / 仓库内完成待外部验收 / 未完成
基线：<开工 commit + 工作树状态>
官方文档：<本任务实际查阅的官方链接，最多 5 个>
改动：<最多 6 条>
测试：<新增或修改的测试及其证明的行为>
验证：<逐条命令 + exit code；不得省略失败>
人工验收：<已完成证据 / 待用户提供资源>
安全检查：<Secret、权限、日志、外部暴露检查>
提交：<commit hash；未提交说明原因>
推送：未推送 / 用户明确授权后已推送
剩余风险：<真实风险；没有则写“无新增已知风险”>
下一任务：<只写编号和名称，不开始执行>
```

现在开始，只执行第一个未完成任务。

## 使用说明（不要一并交给模型也可以）

1. 先把本规划文件和提示词作为独立文档提交保存。
2. 当前 `main` 比 `origin/main` 领先；没有用户明确授权时不要 push。
3. 每轮把“主提示词”完整交给编码模型，它会自动寻找第一个未勾选任务。
4. 每个任务完成后，人工检查提交和报告，再开始下一轮。
5. 到 P5-03、P5-11、P5-12、P5-13、P5-18、P5-21、P5-25 时，重点检查外部资源闸门。
