## 本次唯一目标

只执行 `docs/NEXT_IMPLEMENTATION_PLAN.md` 中第一个未勾选的任务。

如果当前第一个未勾选任务是 `P15-01`，就只做 P15-01。即使时间和上下文充足，也不能继续 P15-02。

## 强制流程

1. 运行 `git status --short --branch` 和 `git log --oneline -5`。
2. 确认工作树干净；如果不干净，先判断是否是用户改动，不得覆盖。
3. 从计划中抄出本任务的“目标、只做、不得做、验证、提交信息”。
4. 按 `CLAUDE.md` 要求查询本任务涉及库的最新官方文档；在汇报中列出查了什么。
5. 只读与本任务直接相关的文件，不做全仓库重构。
6. 若任务是 Bug 或业务逻辑，先写会失败的测试，再修实现。
7. 完成后运行任务列出的全部验证命令；不得省略失败命令。
8. 再运行一次 `git diff --check`，检查 `git diff` 没有无关改动。
9. 在 `docs/NEXT_IMPLEMENTATION_PLAN.md` 将本任务改为 `[x]`，与本任务代码放进同一个提交；不要在计划里伪造未知的提交哈希。
10. 遵守 `CLAUDE.md` 的 Git 规则提交。提交成功后把真实哈希写在最终汇报中，不要 push。
11. 按计划第 9 节固定格式汇报，然后立即停止。

## 禁止事项

- 禁止同时执行两个任务。
- 禁止跳过 Phase 1.5 直接安装 Monaco 或实现 Judge0。
- 禁止把 build 成功描述成业务测试成功。
- 禁止删除、skip 或弱化测试来获得绿色结果。
- 禁止使用 `any`、`@ts-ignore`、空 catch 或硬编码成功响应掩盖问题。
- 禁止修改技术栈和产品范围。
- 禁止 `git reset --hard`、`git clean -fd` 或覆盖用户改动。
- 禁止执行 `git push`。

## 遇到阻塞

只有以下情况可以停止而不提交：

- 缺少用户才能提供的密钥、域名或外部服务权限；
- 当前仓库状态与计划基线冲突且继续会覆盖用户改动；
- 连续三次针对同一根因的安全修复仍失败。

停止时必须给出：准确错误、已运行命令、相关文件、你认为的根因、最小下一步。不要用“环境问题”笼统带过。
